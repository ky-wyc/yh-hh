# QQ 群机器人 MVP 运维说明

## OneBot 登录恢复

MVP 默认使用 NapCat 作为 Linux 云端 OneBot 接入实现。OneBot 容器的数据目录挂载到：

```text
./data/napcat
```

该目录用于保存配置和登录态。更换服务器或恢复服务时，应先恢复该目录，再启动 `onebot` 容器。

NapCat WebUI 默认只绑定服务器本机：

```text
127.0.0.1:6099
```

可以使用 SSH 隧道在本机访问：

```bash
ssh -L 6099:127.0.0.1:6099 <user>@<server-ip>
```

然后打开：

```text
http://127.0.0.1:6099
```

## NapCat 首次登录建议流程

首次部署时建议按以下顺序处理 QQ 小号登录：

1. 启动 `onebot` 容器。
2. 通过 SSH 隧道打开 NapCat WebUI。
3. 按 WebUI 提示扫码登录 QQ 小号。
4. 在 NapCat WebUI 中创建 OneBot v11 WebSocket 客户端，地址填写 `ws://bot-app:8000/onebot/ws`。
5. Token 填写 `.env` 中的 `ONEBOT_ACCESS_TOKEN`。
6. 登录成功后确认后台 OneBot 状态为在线，再到目标群测试 `/ping`。
7. 备份 `data/napcat` 目录。该目录包含登录态，迁移服务器或恢复服务时应和数据库备份一起恢复。

登录态失效时，优先检查 `onebot` 容器日志和 NapCat WebUI。如果日志或 WebUI 提示需要重新验证，按当前 NapCat 版本要求重新登录，再确认 `data/napcat` 目录已更新。不要在未确认登录态的情况下反复重建整个项目目录，否则容易把已有配置和排障线索一起清掉。

## Lagrange 备选说明

仓库仍保留 `deploy/lagrange/appsettings.example.json` 和 `scripts/prepare_lagrange_config.py` 作为 Lagrange 备选。若显式使用 Lagrange，需要提供当前可用的 `LAGRANGE_SIGN_SERVER_URL`，并用：

```bash
ONEBOT_ADAPTER=lagrange sh scripts/deploy_mvp.sh
```

当前默认主线是 NapCat，避免 Lagrange SignServer 不可用导致容器反复重启。

## 反向 WebSocket

Bot Core 暴露：

```text
ws://bot-app:8000/onebot/ws
```

OneBot 实现需要配置为主动连接该地址。生产环境建议配置 `ONEBOT_ACCESS_TOKEN`，并让 NapCat 与 Bot Core 使用同一个 token。Bot Core 支持以下传递方式：

```text
Authorization: Bearer <token>
X-Access-Token: <token>
ws://bot-app:8000/onebot/ws?access_token=<token>
```

## 常见故障

### 服务端口

Docker Compose 默认只把 Bot Core、PostgreSQL 和 Redis 绑定到服务器本机回环地址：

```text
127.0.0.1:8000
127.0.0.1:5432
127.0.0.1:6379
```

这些端口只用于服务器本机排障、备份或 smoke 检查，不应直接对公网开放。`web-admin` 通过 Docker 内部网络代理访问 `bot-app:8000`，OneBot 也通过 Docker 内部网络连接 `ws://bot-app:8000/onebot/ws`。生产环境如果需要远程访问 Bot Core 或数据库，应优先通过 SSH 隧道、内网安全组或专门的反向代理完成，不建议把 `8000`、`5432` 或 `6379` 暴露到公网。

后台默认通过宿主机 `8080` 端口访问。生产环境建议只向可信来源开放该端口，或使用 Nginx、Caddy、云负载均衡等反向代理接入 HTTPS，并配合服务器防火墙、云安全组或 VPN/内网访问控制保护后台入口。

### 后台显示 OneBot 离线

检查项：

- `bot-app` 是否运行。
- `onebot` 是否运行。
- OneBot 反向 WebSocket 地址是否为 `ws://bot-app:8000/onebot/ws`。
- OneBot 登录态是否过期。
- `ONEBOT_ACCESS_TOKEN` 是否一致。

后台 OneBot 在线状态以当前反向 WebSocket 连接为准。OneBot 快速重连时，旧连接的延迟断开不会覆盖新连接状态；如果后台持续离线，应优先查看 `onebot` 容器日志和 Bot Core 最近错误。

### 群里没有回复

检查项：

- 群是否在 `ALLOWED_GROUPS` 中。
- 后台群开关是否启用。
- 回复模式是否允许命令、@ 回复或 active 疑问句回复。
- 是否触发了限流。
- `/api/system/logs` 中是否有丢弃原因。

白名单外群的消息只会记录丢弃原因，不会自动出现在后台群管理列表。需要启用新群时，先把群号加入 `ALLOWED_GROUPS`，再在后台手动添加或等待该群首次文本消息创建配置。

### 基础群管配置

后台群详情中可以配置新人欢迎、刷屏检测和阶梯处罚，也可以先套用“观察模式、宽松、标准、严格”群管策略模板后再保存。刷屏检测命中后会写入 `flood_mute` 审计；管理员执行 `/warn`、`/mute` 也会写入审计。近期违规统计基于这些审计记录计算，不需要额外数据表。

建议起步配置：

- `累计窗口小时`：24。
- `阶梯处罚`：先关闭观察，确认误判率低后再打开。
- `处罚倍率`：2。
- `禁言封顶秒`：3600。

打开阶梯处罚后，同一用户在累计窗口内重复触发刷屏，禁言时长会按 `基础禁言秒数 * 倍率^(近期违规次数)` 增加，并受封顶限制。后台群详情会展示近期违规用户、累计次数和最近时间。

需要追踪单个用户或某类群管操作时，到日志中心的“审计日志”页按群号、操作、目标类型和目标 ID 筛选。例如查看某个用户近期违规，可以筛选 `目标类型=user` 和目标 QQ；查看自动刷屏处罚，可以筛选 `操作=flood_mute`。

### 先不接真实 QQ，验证 Bot Core

可以先执行 HTTP 冒烟检查：

```bash
python scripts/smoke_check.py --base-url http://127.0.0.1:8000 --username "$ADMIN_USERNAME" --password "$ADMIN_PASSWORD" --expect-cache-backend redis
```

需要留存排障材料时，可以输出 JSON 报告：

```bash
python scripts/smoke_check.py --base-url http://127.0.0.1:8000 --username "$ADMIN_USERNAME" --password "$ADMIN_PASSWORD" --expect-cache-backend redis --report-file data/smoke-report.json
```

如果 NapCat 已经完成登录并连接 Bot Core，还可以追加 OneBot 在线强校验：

```bash
python scripts/smoke_check.py --base-url http://127.0.0.1:8000 --username "$ADMIN_USERNAME" --password "$ADMIN_PASSWORD" --expect-cache-backend redis --require-onebot-online
```

部署脚本也支持同样的生产强校验开关。真实 OneBot 登录后可使用：

```bash
STRICT_PREFLIGHT=1 REQUIRE_ONEBOT_ONLINE=1 sh scripts/deploy_mvp.sh
```

真实 QQ 群完成收发测试和群管命令测试后可使用：

```bash
STRICT_PREFLIGHT=1 REQUIRE_ONEBOT_ACTIVITY=1 REQUIRE_MVP_CORE_LOGS=1 REQUIRE_ADMIN_LITE_AUDIT=1 sh scripts/deploy_mvp.sh
```

真实 QQ 群完成 `/ping`、`/help`、`/dice`、`/ai` 或 @ 回复测试后，可以追加 `--require-onebot-activity`，确认后台已记录最近入站事件和出站动作：

```bash
python scripts/smoke_check.py --base-url http://127.0.0.1:8000 --username "$ADMIN_USERNAME" --password "$ADMIN_PASSWORD" --expect-cache-backend redis --require-onebot-online --require-onebot-activity
```

真实核心命令完成后，可以追加消息日志强校验，确认后台消息日志已经记录 `/ping`、`/help`、`/dice`、`/ai` 的处理结果：

```bash
python scripts/smoke_check.py --base-url http://127.0.0.1:8000 --username "$ADMIN_USERNAME" --password "$ADMIN_PASSWORD" --expect-cache-backend redis --require-mvp-core-logs
```

真实群管命令完成后，可以追加 admin-lite 审计强校验，确认后台审计日志已经记录 `warn` 和 `banword_add`：

```bash
python scripts/smoke_check.py --base-url http://127.0.0.1:8000 --username "$ADMIN_USERNAME" --password "$ADMIN_PASSWORD" --expect-cache-backend redis --require-admin-lite-audit
```

再模拟 OneBot 反向 WebSocket。若真实 NapCat 已经在线，先临时停止 `onebot` 容器，避免模拟 WebSocket 临时占用 Bot Core 的当前 OneBot 连接状态：

```bash
python scripts/onebot_simulate.py --ws-url ws://127.0.0.1:8000/onebot/ws --group-id 10001 --access-token "$ONEBOT_ACCESS_TOKEN"
```

模拟成功时会看到 `send_group_msg` 动作，消息内容为 `pong`。

### 大模型不可用

检查项：

- 后台是否配置 API Key。
- `LLM_BASE_URL` 是否是 OpenAI-compatible 接口。
- 模型名是否正确。
- `/api/settings/llm/test` 是否返回失败原因。

### 知识库 Embedding 不可用

默认 `EMBEDDING_PROVIDER=local`，不需要外部 API Key，也适合小规模知识库和离线测试。需要接入中转站或外部 embedding 模型时，在后台“模型与 API 设置”里的“知识库 Embedding 设置”切换为 `openai_compatible`，填写 Base URL、API Key、模型名和向量维度。

注意：

- `EMBEDDING_API_KEY` 不要提交到 GitHub，生产环境优先通过后台或 `.env` 配置。
- 切换 provider、模型或向量维度后，需要在知识库页面对已有文档执行“重建索引”。
- 外部 embedding 建索引失败时，知识库文档会显示 `index_status=failed` 和 `index_error`。
- 如果外部 embedding 查询临时失败，检索会保留关键词分数兜底，但语义召回效果会下降。
- 后台知识库页面支持按群号和索引状态筛选，并提供“重建当前筛选”和“重试失败”。小规模知识库可以直接使用该同步批处理；大规模知识库后续再升级为异步队列。

## 备份

执行：

```bash
sh scripts/backup.sh
```

备份文件输出到：

```text
./data/backups
```

生产环境除了 PostgreSQL 备份外，还应定期备份 `data/napcat` 登录态目录和 `.env` 配置文件。`data/napcat` 和 `.env` 均包含敏感信息，备份文件需要限制访问权限。

备份和恢复脚本要求 `POSTGRES_USER`、`POSTGRES_DB` 只包含字母、数字和下划线。该限制用于保证 shell 脚本和 SQL 恢复流程稳定；如需使用更复杂的数据库名称，应先扩展并重新验证备份恢复脚本。

## 备份恢复演练

不想立刻执行真实恢复时，可以先生成非破坏性演练报告。该命令只检查 `.env`、备份文件、NapCat 登录态目录和恢复命令，不会停止服务、不会重建数据库、不会导入备份：

```bash
python scripts/backup_restore_drill.py \
  --env-file .env \
  --report-file data/backup-restore-drill.json
```

如果要指定某个备份文件：

```bash
python scripts/backup_restore_drill.py \
  --env-file .env \
  --backup-file ./data/backups/postgres-YYYYmmdd-HHMMSS.sql \
  --report-file data/backup-restore-drill.json
```

报告中 `status=passed` 表示恢复演练前置条件满足。报告里的 `restore_executed` 固定为 `false`，用于区分“演练检查”与“真实恢复”。建议留存以下证据：

- `data/backup-restore-drill.json`。
- `sh scripts/backup.sh` 的输出。
- 备份文件路径、大小和权限检查结果。
- 真实恢复演练后的 `smoke_check.py` 输出。

## 恢复

恢复 PostgreSQL 前，先确认目标备份文件存在，并尽量在业务低峰执行。恢复脚本会停止 `bot-app`、`web-admin` 和 `onebot`，重建 PostgreSQL 数据库，再启动服务：

```bash
sh scripts/restore_postgres.sh ./data/backups/postgres-YYYYmmdd-HHMMSS.sql
```

非交互环境可以使用：

```bash
FORCE=1 sh scripts/restore_postgres.sh ./data/backups/postgres-YYYYmmdd-HHMMSS.sql
```

恢复后建议立即执行：

```bash
python scripts/smoke_check.py --base-url http://127.0.0.1:8000 --username "$ADMIN_USERNAME" --password "$ADMIN_PASSWORD" --expect-cache-backend redis
```

如果是迁移服务器，还需要同步恢复 `.env` 和 `data/napcat` 登录态目录，再启动 `onebot` 并确认后台 OneBot 状态恢复在线。
