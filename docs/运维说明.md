# QQ 群机器人 MVP 运维说明

## OneBot 登录恢复

MVP 默认使用 Lagrange.OneBot 作为 Linux 云端接入实现。OneBot 容器的数据目录挂载到：

```text
./data/onebot
```

该目录用于保存配置和登录态。更换服务器或恢复服务时，应先恢复该目录，再启动 `onebot` 容器。

本仓库提供了一个反向 WebSocket 配置样例：

```text
deploy/lagrange/appsettings.example.json
```

部署时可以自动生成：

```bash
python scripts/prepare_lagrange_config.py --env-file .env
```

输出文件：

```text
data/onebot/appsettings.json
```

脚本会读取 `.env` 中的 `BOT_QQ`、`ONEBOT_REVERSE_WS_URL`、`ONEBOT_ACCESS_TOKEN`，写入 `Account` 和 `ReverseWebSocket` 配置。生成后仍需按实际 QQ 小号和 Lagrange.OneBot 版本确认登录方式。Lagrange.OneBot 的具体配置字段以官方文档为准：https://lagrangedev.github.io/Lagrange.Doc/v1/Lagrange.OneBot/

## Lagrange 首次登录建议流程

首次部署时建议按以下顺序处理 QQ 小号登录：

1. 先确认 `.env` 中 `BOT_QQ`、`ONEBOT_REVERSE_WS_URL`、`ONEBOT_ACCESS_TOKEN` 已配置。
2. 执行 `python scripts/prepare_lagrange_config.py --env-file .env` 生成 `data/onebot/appsettings.json`。
3. 启动服务后查看 `onebot` 容器日志，按日志提示完成扫码、验证码或设备验证。
4. 登录成功后确认后台 OneBot 状态为在线，再到目标群测试 `/ping`。
5. 备份 `data/onebot` 目录。该目录包含登录态，迁移服务器或恢复服务时应和数据库备份一起恢复。

登录态失效时，优先检查 `onebot` 容器日志。如果日志提示需要重新验证，先停止 `onebot` 容器，按当前 Lagrange.OneBot 版本要求重新登录，再确认 `data/onebot` 目录已更新。不要在未确认登录态的情况下反复重建整个项目目录，否则容易把已有配置和排障线索一起清掉。

## 反向 WebSocket

Bot Core 暴露：

```text
ws://bot-app:8000/onebot/ws
```

OneBot 实现需要配置为主动连接该地址。生产环境建议配置 `ONEBOT_ACCESS_TOKEN`，并让 Lagrange.OneBot 与 Bot Core 使用同一个 token。Bot Core 支持以下传递方式：

```text
Authorization: Bearer <token>
X-Access-Token: <token>
ws://bot-app:8000/onebot/ws?access_token=<token>
```

## 常见故障

### 服务端口

Docker Compose 默认只把 Bot Core、PostgreSQL 和 Redis 绑定到服务器本机回环地址：

```text
127.0.0.1:8000
127.0.0.1:5432
127.0.0.1:6379
```

这些端口只用于服务器本机排障、备份或 smoke 检查，不应直接对公网开放。`web-admin` 通过 Docker 内部网络代理访问 `bot-app:8000`，OneBot 也通过 Docker 内部网络连接 `ws://bot-app:8000/onebot/ws`。生产环境如果需要远程访问 Bot Core 或数据库，应优先通过 SSH 隧道、内网安全组或专门的反向代理完成，不建议把 `8000`、`5432` 或 `6379` 暴露到公网。

后台默认通过宿主机 `8080` 端口访问。生产环境建议只向可信来源开放该端口，或使用 Nginx、Caddy、云负载均衡等反向代理接入 HTTPS，并配合服务器防火墙、云安全组或 VPN/内网访问控制保护后台入口。

### 后台显示 OneBot 离线

检查项：

- `bot-app` 是否运行。
- `onebot` 是否运行。
- OneBot 反向 WebSocket 地址是否为 `ws://bot-app:8000/onebot/ws`。
- OneBot 登录态是否过期。
- `ONEBOT_ACCESS_TOKEN` 是否一致。

后台 OneBot 在线状态以当前反向 WebSocket 连接为准。OneBot 快速重连时，旧连接的延迟断开不会覆盖新连接状态；如果后台持续离线，应优先查看 `onebot` 容器日志和 Bot Core 最近错误。

### 群里没有回复

检查项：

- 群是否在 `ALLOWED_GROUPS` 中。
- 后台群开关是否启用。
- 回复模式是否允许命令、@ 回复或 active 疑问句回复。
- 是否触发了限流。
- `/api/system/logs` 中是否有丢弃原因。

白名单外群的消息只会记录丢弃原因，不会自动出现在后台群管理列表。需要启用新群时，先把群号加入 `ALLOWED_GROUPS`，再在后台手动添加或等待该群首次文本消息创建配置。

### 先不接真实 QQ，验证 Bot Core

可以先执行 HTTP 冒烟检查：

```bash
python scripts/smoke_check.py --base-url http://127.0.0.1:8000 --username "$ADMIN_USERNAME" --password "$ADMIN_PASSWORD" --expect-cache-backend redis
```

需要留存排障材料时，可以输出 JSON 报告：

```bash
python scripts/smoke_check.py --base-url http://127.0.0.1:8000 --username "$ADMIN_USERNAME" --password "$ADMIN_PASSWORD" --expect-cache-backend redis --report-file data/smoke-report.json
```

如果 Lagrange.OneBot 已经完成登录，还可以追加 OneBot 在线强校验：

```bash
python scripts/smoke_check.py --base-url http://127.0.0.1:8000 --username "$ADMIN_USERNAME" --password "$ADMIN_PASSWORD" --expect-cache-backend redis --require-onebot-online
```

部署脚本也支持同样的生产强校验开关。真实 OneBot 登录后可使用：

```bash
STRICT_PREFLIGHT=1 REQUIRE_ONEBOT_ONLINE=1 sh scripts/deploy_mvp.sh
```

真实 QQ 群完成收发测试和群管命令测试后可使用：

```bash
STRICT_PREFLIGHT=1 REQUIRE_ONEBOT_ACTIVITY=1 REQUIRE_MVP_CORE_LOGS=1 REQUIRE_ADMIN_LITE_AUDIT=1 sh scripts/deploy_mvp.sh
```

真实 QQ 群完成 `/ping`、`/help`、`/dice`、`/ai` 或 @ 回复测试后，可以追加 `--require-onebot-activity`，确认后台已记录最近入站事件和出站动作：

```bash
python scripts/smoke_check.py --base-url http://127.0.0.1:8000 --username "$ADMIN_USERNAME" --password "$ADMIN_PASSWORD" --expect-cache-backend redis --require-onebot-online --require-onebot-activity
```

真实核心命令完成后，可以追加消息日志强校验，确认后台消息日志已经记录 `/ping`、`/help`、`/dice`、`/ai` 的处理结果：

```bash
python scripts/smoke_check.py --base-url http://127.0.0.1:8000 --username "$ADMIN_USERNAME" --password "$ADMIN_PASSWORD" --expect-cache-backend redis --require-mvp-core-logs
```

真实群管命令完成后，可以追加 admin-lite 审计强校验，确认后台审计日志已经记录 `warn` 和 `banword_add`：

```bash
python scripts/smoke_check.py --base-url http://127.0.0.1:8000 --username "$ADMIN_USERNAME" --password "$ADMIN_PASSWORD" --expect-cache-backend redis --require-admin-lite-audit
```

再模拟 OneBot 反向 WebSocket。若真实 Lagrange.OneBot 已经在线，先临时停止 `onebot` 容器，避免模拟连接干扰当前 OneBot 在线状态：

```bash
python scripts/onebot_simulate.py --ws-url ws://127.0.0.1:8000/onebot/ws --group-id 10001 --access-token "$ONEBOT_ACCESS_TOKEN"
```

模拟成功时会看到 `send_group_msg` 动作，消息内容为 `pong`。

### 大模型不可用

检查项：

- 后台是否配置 API Key。
- `LLM_BASE_URL` 是否是 OpenAI-compatible 接口。
- 模型名是否正确。
- `/api/settings/llm/test` 是否返回失败原因。

## 备份

执行：

```bash
sh scripts/backup.sh
```

备份文件输出到：

```text
./data/backups
```

生产环境除了 PostgreSQL 备份外，还应定期备份 `data/onebot` 登录态目录和 `.env` 配置文件。`data/onebot` 和 `.env` 均包含敏感信息，备份文件需要限制访问权限。

备份和恢复脚本要求 `POSTGRES_USER`、`POSTGRES_DB` 只包含字母、数字和下划线。该限制用于保证 shell 脚本和 SQL 恢复流程稳定；如需使用更复杂的数据库名称，应先扩展并重新验证备份恢复脚本。

## 恢复

恢复 PostgreSQL 前，先确认目标备份文件存在，并尽量在业务低峰执行。恢复脚本会停止 `bot-app`、`web-admin` 和 `onebot`，重建 PostgreSQL 数据库，再启动服务：

```bash
sh scripts/restore_postgres.sh ./data/backups/postgres-YYYYmmdd-HHMMSS.sql
```

非交互环境可以使用：

```bash
FORCE=1 sh scripts/restore_postgres.sh ./data/backups/postgres-YYYYmmdd-HHMMSS.sql
```

恢复后建议立即执行：

```bash
python scripts/smoke_check.py --base-url http://127.0.0.1:8000 --username "$ADMIN_USERNAME" --password "$ADMIN_PASSWORD" --expect-cache-backend redis
```

如果是迁移服务器，还需要同步恢复 `.env` 和 `data/onebot` 登录态目录，再启动 `onebot` 并确认后台 OneBot 状态恢复在线。
